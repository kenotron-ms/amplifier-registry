The Solution

A community registry. Zero backend. CI is the backend.

Single HTML file and two JSON files on GitHub Pages. No server, no database. A GitHub Actions pipeline sweeps, scores, and embeds every 2 hours.

# GitHub Actions (every 2 hours) sweep.mjs → GitHub API → stars, forks, behaviors/ score.mjs → Claude → 5-dimension quality rubric embed.mjs → MiniLM-L6 → 384-dim semantic vectors git commit → bundles.json + embeddings.json
1 HTML file, zero build step
2 JSON files (bundles + embeddings)
2h CI cycle, 12x more frequent than nightly
42 bundles 13 contributors 3.2k total stars

Key point: The entire backend is a GitHub Actions cron job. Three scripts run every 2 hours: sweep discovers bundles and scans their behaviors, score sends each README to Claude for quality rating, and embed produces 384-dimensional vectors using the same model the browser uses for queries. The output is two JSON files committed to the repo, which GitHub Pages serves as a static site.

If challenged: "Why not use a real backend with a database?" Because the data changes slowly (bundles don't update every minute) and the query model is simple (search + filter). A static site with pre-computed data is simpler to operate, costs nothing, has no uptime risk, and deploys automatically on every commit. The 2-hour refresh is more than sufficient for a registry of this size.

Transition: That's the architecture. Now let me show you the first thing it enables: natural language search that works on the first keystroke.

Natural language search. First keystroke instant.

After

Pre-computed embeddings 130 KB

CI pre-computes 384-dim vectors for all 42 bundles (all-MiniLM-L6-v2). Same model runs in CI and browser. Vectors share the same embedding space. First keystroke is instant.

CI produces the hard part. Browser just computes the query vector.

Key point: The old approach loaded a 6MB ML model and computed embeddings for all 42 bundles in the browser on every page visit. That meant a multi-second "Indexing bundles..." delay before search worked. The fix was to move the expensive computation to CI: pre-compute 384-dimensional semantic vectors using all-MiniLM-L6-v2 and store them as a 130KB JSON file. The browser downloads that JSON and only needs to embed the user's search query, which is a single vector operation.

If challenged: "If the model runs in the browser via WASM, isn't it still slow?" The model load is lazy and cached by the browser. On first search, there's a brief model initialization. But the critical difference is that the browser doesn't need to embed 42 bundle descriptions anymore, just one query string. That takes milliseconds, not seconds. On repeat visits, the model is cached and search is truly instant.

Transition: Semantic search makes bundles findable. But users don't always want the whole bundle. The next feature goes deeper: behavior-level discovery.

Confidence: High. The 6MB model size is from the Transformers.js package. The 130KB embeddings.json is measured from the committed file. The "same embedding space" claim is verified: both CI and browser use the all-MiniLM-L6-v2 model via Transformers.js.

Feature · Behavior-Level Discovery

Don't install the whole bundle. Install just the capability you need.

Install a bundle → Install exactly the capability I want
$ amplifier bundle add \ git+https://github.com/AmpDaily/foundation-expert@main \ #subdirectory=behaviors/logging.yaml \ --app
28 / 42 Bundles with scanned behaviors (66%)
12 Behaviors in foundation-expert, the most of any bundle

Key point: Bundles are monolithic by default: install everything or nothing. But each repo has a behaviors/ directory with individual YAML files that define focused capabilities. The sweep now walks every repo's git tree, finds all behaviors/*.yaml, parses name, description, tools, hooks, and agents, and generates a ready-to-run install command for each one. Users can now install a single behavior instead of the entire bundle.

If challenged: "Why not just document this in each bundle's README?" Because the install command requires a specific git URL format with a subdirectory fragment. Nobody is going to construct that by hand. Generating the command and providing a click-to-copy button removes all friction. The sweep also surfaces behaviors that README authors forgot to document.

Transition: Behavior scanning was the hard part. The next step was making behaviors visible without clicking into each bundle. It took 13 lines of code.

Confidence: High. 28 of 42 bundles confirmed from the latest sweep data. foundation-expert's 12 behaviors verified from the behaviors/ directory in that repo.

Architecture

No server. No database. Three JSON files and a cron job.

Key point: The entire system is three scripts, two JSON files, one HTML file, and a cron job. sweep.mjs hits the GitHub API to discover bundles, collect stars and forks, and scan each repo's behaviors/ directory. score.mjs sends each bundle's README to Claude and gets a quality rating on a 5-dimension rubric. embed.mjs runs all-MiniLM-L6-v2 to produce 384-dimensional semantic vectors for search. The output is committed as static JSON and auto-deployed to GitHub Pages. The browser (index.html, 1,663 lines) loads the JSON and uses Transformers.js for query embedding.

If challenged: "What happens if the GitHub Actions cron fails?" The site keeps serving the last successful sweep. Stale data is better than no data, and the 2-hour cycle means the worst-case staleness is 2 hours. We also get email alerts on workflow failures. There's no single point of failure that takes the registry offline.

Transition: This architecture was not designed in advance and implemented. It was built iteratively. And every commit that built it has the same co-author.

Confidence: High. Every component described here is directly verifiable in the repository. Line count from wc -l. Script names and their dependencies are from the workflow YAML.
